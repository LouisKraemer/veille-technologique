Avenir des applications natives

Introduction

En 1994, IBM sort le "SIMON". Ce périphérique est considéré comme le premier permettant une inetraction poussée avec l'utilisateur. La première application mobile est née. En 2007, Apple frappe un grand coup en lançant le premier IPhone. Celui-ci dispose d'un écran multitouch de 3.5 pouces et embarque une dizaine d'application (SMS, appels, calendrier, ...). C'est la première fois que les applications mobiles sont disponibles à grande échelle. Au fur et à mesure des années, ces applications se sont développées et sont devenues indispensables. Aujourd'hui, les entreprises cherchent à ceux que leurs applications apparaissent comme une extension fluide de leur site ou de leur ecosystème.

Le problème actuel réside dans le schéma ci-dessus. Les applications sont, à la base, développées en natif. Cela signifie que pour une unique idée, chaque système d'exploitation mobile va nécessiter son propre développement (en Java pour Android, en Objective C ou Swift pour iOS). Il y a donc une difficulté non seulement dans l'apprentissage de multiples langages mais aussi dans le maitien de l'application qui sera une tâche lourde en temps de travail.

Néanmoins, une alternative à fait son apparition : les applications hybrides. Celles-ci se basent sur le développement Web pour créer des applications mobiles. Mais elles ne sont pas apparues tout de suite.

Applications Web Progressives

L'arrivée de nouveaux acteurs du Web a permis de nombreux changements dans le milieu des applications mobiles.

Tout d'abord, les applications webs progressives sont apparues. Le terme a été initialement émis en 2015 par deux ingénieurs de Google. Le buts était de rendre l'expérience utilisateur d'un site aussi proche que possible de celle offerte par une application native. Pour ce faire, plusieurs critères devait être remplis.

Service Workers

On peut comparer le comportement d'un service worker à celui d'un serveur proxy. Le but de celui-ci est de permettre un bon fonctionnement sans tenir compte des conditions réseaux (que le signal soit faible ou même absent). Il permet à l'utilisateur "d'oublier" qu'il se trouve sur un site web. Grâce à ces service workers, le site peut utiliser d'autres fonctionnalités comme les notifications.

Notifications Push

Les notifications push permettent d'informer l'utilisateur d'une mise à jour sur le site consulté même en arrière plan. Ainsi ce dernier reste au courant de l'actualité et des modifications. Ce comportement peut s'observer lors d'une visite sur le site mobile du réseau social Facebook. La première fois, une bannière demandant l'accès aux notifications sera présentée. Si l'on répond oui, Facebook se servira du navigateur pour nous prévenir des actualités de notre compte.

Fluidité

La fluidité est une barrière importante qui a longtemps séparé les applications natives du web. Les sites ont souvent été plus lent et victime de ralentissement. Les applications webs progressives mettent fin à cela en permettant, sur les périphériques adaptés, d'afficher 60 images par seconde ce qui est considéré comme suffisant pour avoir la meilleure expérience utilisateur possible.

Responsive Design

Le responsive design est de plus en plus utilisé pour obtenir un but précis : délivrer la même expérience peu importe la taille de l'écran du périphérique utilisé (TV, ordinateur fixe, ordinateur portable, tablette, téléphone, ...). Twitter a largement facilité cette tâche en créant le framework Bootstrap en 2010. Avec son système de grille, un développeur peut créer un site qui s'adaptera à chaque taille d'écran afin que l'utilisateur ne soit jamais déconcerté.

Ajout à l'écran d'accueil

La plus grande différence qui réside, au premier abord entre un site web et une application et la façon d'ouvrir celle-ci. Les applications webs progressives proposent donc une fonction "Ajouter à l'écran d'accueil" qui permet la création d'un icône sur le lanceur d'application du périphérique. Ainsi, un simple clique lancera le site et le confondra quasi totalement avec une application native.

Applications hybrides

Principe

Le principe est d'encasuler une application Web au sein d'un container émulant un navigateur Web. Ce container sera lancé de la même manière qu'une application classique.

On constate ici qu'il y a trois parties :

  • La partiee rouge représente l'application. Elle contient le code visuel et la logique du site.
  • Les parties violettes contiennent le code source du site (HTML, CSS, JS), l'accès au matériel (capteurs, caméra, ...).
  • Le système d'exploitation du périphérique, en vert.

Les parties rouge et violettes seront encapsulée dans une enveloppe native et le tout pourra être installé de façon classique.



Acteurs principaux

Il existe de multiple acteurs dans le monde des applications hybrides. Nous nous intéresserons aux deux majeurs.

Cordova

En 2009, Apache reprend le projet PhoneGap et le nomme Cordova. Ce projet a pour but de permettre la création d'application mobile native via du code Web (HTML, CSS, JS). Cordova dispose d'API qui communique entre le code JavaScript et les différents capteurs du périphériques (GPS, caméra, ...). On peut ainsi, à l'aide de code Web, manipuler le matériel et connaître l'environnement extérieur à l'application.

On peut voir sur ce schema que les API font office de pont entre la partie Web et la partie matériel. Une fois que cela est effectué, l'ensemble du code sera visualisable au travers d'une WebView. C'est un container faisant office de navigateur intégré. L'utilisateur pourra alors ouvrir l'application et manipuler la quasi totalité de ce qu'une application native peut manipuler.

Ionic

Ionic est un framework JavaScript open-source créé en 2013 par Max Lynch, Ben Sperry, et Adam Bradley. Il se base sur le framework Angular créé par Google et reconnu par de nombreux développeurs. Le point important d'Ionic est sa librairie HTML5 qui comprend une multitude d'éléments et de transitions très semblables aux éléments natifs d'Android et iOS.

On constate dans cette application de chat que la ressemblance est frappante. L'interface est quasi identique à celle de messenger (chat Facebook) d'il y a deux ans. On se rapproche ainsi fortement d'une application native.

Application Cross-platform

Principe

Il ne s'agit plus d'utiliser une interface Web mais de se servir des composants du systèmes d'exploitations pour animer l'interface. Il n'y aura donc plus de différence aussi bien visuel qu'interne par rapport à une implémentation native.

L'application cross-platform ne se sert pas d'éléments HTML pour son interface, la structure est semblable à celle d'une application hybride mais la seule partie web du code et le Javascript. On va donc placer différents éléments natifs (boutonn, champs de recherche, slider, ...) via du code JS et les faire interagir entre eux. Cette solution présente plusieurs avantages :

  • Aucune différence avec une application native en terme de design
  • Performances supérieures à celle d'une solution hybride
  • Possibilité d'intégrer du code natif au sein de l'application

Acteur

Facebook développe et inclut sur sa page web, en 2011, un Framework JavaScript nommé React (souvent comparé à Angular de Google). En 2013, lors d'un hackathon interne à l'entreprise, une équipe de développeur mette au point un nouveau Framework : React Native.
Ce framework est vite adopté par de nombreuses firmes pour toutes les raisons énoncées précédemment, malgré son apprentissage plus difficile que les framework hybride.

On voit que des entreprises comme Skype, AirB&b ou encore Instagram ont fait confiance à React Native et leurs applications ne peuvent pas être discerné d'une application native.

Conclusion

Web Progressives Hybrides/Cross-platform Natives
Accessibilité Très accessible (site) Via les stores Via les stores
Développement Code Web Code Web + code natif (React Native) Java (Android) et Objective C ou Swift (iOS)
Accès matériel Limité Quasi total Total
Performances Faibles Variables (Ionic < React Native) Fortes

On peut voir les différences entres ces technologies. Néanmoins, les applications natives ne sont pas enterrées pour autant. Si l'objectif d'un projet et d'avoir une interface simple avec peu d'animations (flux d'actualités), les applications webs progressives sont une des meilleures soltuions. Dans le cas d'une applications plus complexes avec besoin d'accéder au matériel du périphérique, beaucoup d'interactions possibles, il faudra se tourner vers une application hybride ou cross-platform. Beaucoup de sociétés font comme cela car cela diminue le temps de développement et facilite le maintien. Enfin, pour une application gourmande en ressource (jeux mobile 3D), la solution native est la seule viable.

Pour cela, les applications natives sont encore bien implantées dans le marchés des applications mobiles et, à moins d'un nouveau framework, graderont leur place.
